On structures like maps or \mbox{\hyperlink{doc_decisions_5_partially_implemented_array_md}{arrays}} there are different possibilities which keys are looked up in the Key\+Set and which are simply iterated.

Without any guidelines, applications would provide arbitrary inconsistent behavior.\hypertarget{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2415}{}\doxysection{Constraints}\label{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2415}

\begin{DoxyItemize}
\item Applications that have good reasons to ignore the guidelines (e.\+g. they only read from one namespace), are allowed to do so.
\end{DoxyItemize}\hypertarget{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2417}{}\doxysection{Considered Alternatives}\label{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2417}

\begin{DoxyItemize}
\item only lookup the roots and then iterate over the next keys
\item let the applications do what they want without any guideline
\end{DoxyItemize}\hypertarget{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2418}{}\doxysection{Decision}\label{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2418}
Every key that an application wants to use, must be looked up with {\ttfamily ks\+Lookup} using a cascading lookup key.\hypertarget{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2419}{}\doxysection{Rationale}\label{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2419}

\begin{DoxyItemize}
\item very simple rule, easy to understand, easy to follow
\item provides consistent behavior ({\ttfamily spec} is always honored)
\item {\ttfamily ks\+Lookup} is quite cheap as it has only a few simple loops, only one allocation and less than 10\% of CPU time in profiling, even in very simple applications with many lookups.
\end{DoxyItemize}\hypertarget{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2420}{}\doxysection{Implications}\label{doc_decisions_6_implemented_lookup_every_key_md_autotoc_md2420}
