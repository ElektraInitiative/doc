There are ambiguous cases where the same return value can refer to multiple problems\+:


\begin{DoxyItemize}
\item name modifications which can be either invalid name or locking the key name
\item getting values of (non-\/)binary keys
\end{DoxyItemize}\hypertarget{doc_decisions_error_handling_md_autotoc_md1844}{}\doxysection{Constraints}\label{doc_decisions_error_handling_md_autotoc_md1844}
We also have to consider that many parts of the API very constrained in what values they can use for error codes. If a function returns a pointer the only possible error value by definition is NULL.\hypertarget{doc_decisions_error_handling_md_autotoc_md1845}{}\doxysection{Assumptions}\label{doc_decisions_error_handling_md_autotoc_md1845}

\item Add second channel for getting information about errors
\item Return error codes directly from functions where failures are expected, e.\+g. {\ttfamily kdb\+Get}, {\ttfamily key\+Set\+Name}
\item Harmonize return values from all functions and move error reporting to second channel
