The entire logic for backends has been rewritten, to allow for more flexibility und an unlimited number of plugins. Instead of calling plugins directly, {\ttfamily libelektra-\/kdb} now only calls so-\/called backend plugins and special hook plugins. There is a \mbox{\hyperlink{doc_dev_backend-plugins_md}{contract}} between {\ttfamily libelektra-\/kdb} and the backend plugins. All backend plugins must adhere to this contract. To achieve this goal, most backend plugins will call other plugins (like {\ttfamily libelektra-\/kdb} did previously).

The logic previously implemented in {\ttfamily libelektra-\/kdb} was moved to the new default backend plugin {\ttfamily backend}. It works like the old system, but now also allows an unlimited number of plugins in positions where that makes sense. For example, you can have unlimited {\ttfamily postgetstorage} plugins, but only a single {\ttfamily getresolver}.

There have also been slight changes to {\ttfamily kdb\+Get} and {\ttfamily kdb\+Set}. Please read their API docs to find out, if you rely on any behavior that has been altered. You can also read the \mbox{\hyperlink{doc_dev_kdb-operations_md}{new low-\/level docs}} to find out all the intricate details.

The structure of {\ttfamily system\+:/elektra/mountpoints} changed as well. Take a look at the \mbox{\hyperlink{doc_dev_mountpoints_md}{new docs}}, if you need to know details.\hypertarget{doc_news_2023-03-03_0_9_12_md_autotoc_md4056}{}\doxysubsubsection{Updating config}\label{doc_news_2023-03-03_0_9_12_md_autotoc_md4056}
The mountpoint configuration format contains {\bfseries{breaking changes}} and a manual upgrade process is needed. Follow these steps to upgrade the old mountpoint configuration to the new format\+:

\begin{quote}
{\bfseries{Warning}}\+: BACK UP YOUR CONFIG FILES BEFORE UPDATING! We recommend making a backup of the file printed by {\ttfamily kdb file system\+:/elektra/mountpoints} before updating your installation. In the unlikely case that the migration script fails, you can still use the information from the backup to manually recreate your mountpoints. \end{quote}
To update your existing {\ttfamily system\+:/elektra/mountpoints} data you can use the migration script.

\begin{quote}
{\bfseries{Note}}\+: To run the script you must have Elektra, Python ($>$= 3.\+7) and the Python binding installed. The script uses the Python binding to manipulate {\ttfamily Key}s and {\ttfamily Key\+Set}s, but it does not use the {\ttfamily kdb} CLI tool, or the {\ttfamily KDB} API. It is safe to run this script before, or after you update your Elektra installation. \end{quote}
By default, the script loads the file {\ttfamily /etc/kdb/elektra.ecf}. If you changed where {\ttfamily system\+:/elektra/mountpoints} is stored, you can provide an alternative path\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{./migrate-\/mountpoints.py /path/to/your/mountpoints/config.file}

\end{DoxyCode}


\begin{quote}
{\bfseries{Note}}\+: Because the script does not use the {\ttfamily KDB} API it only works, if the mountpoints config file uses the default {\ttfamily dump} format.

If your mountpoints config file is not using the {\ttfamily dump} format, you may still be able to use the migration script. However, in that case, you will have to use the script {\bfseries{before}} updating your Elektra installation\+:


\begin{DoxyEnumerate}
\item Run {\ttfamily kdb export system\+:/elektra/mountpoints dump} to get a copy of your mountpoints config in {\ttfamily dump} format
\item Write this data to a file and run the migration script on the file.
\item To get the data back in your original format you can use
\end{DoxyEnumerate}


\begin{DoxyCode}{0}
\DoxyCodeLine{./migrate-\/mountpoints.py /path/to/file/from/step2 | kdb convert dump your-\/format > /path/to/converted/file}

\end{DoxyCode}



\begin{DoxyEnumerate}
\item Run {\ttfamily kdb file system\+:/elektra/mountpoints} to find out where your mountpoint config is stored. Make sure to back up this file, before upgrading your installation.
\item Now upgrade your Elektra installation.
\item Copy the file {\ttfamily /path/to/converted/file} from step 3 to the location you got in step 4. 
\end{DoxyEnumerate}\end{quote}
The script will read the old mountpoint configuration from the given file. It will convert the configuration and print the new version to {\ttfamily stdout}.

You can inspect the output to make sure, everything is in order. When you are ready to commit the changes, you can manually edit the config file, or use\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{./migrate-\/mountpoints.py -\/-\/output=/etc/kdb/elektra.ecf /etc/kdb/elektra.ecf}

\end{DoxyCode}

Thanks to \+\_\+(Maximilian Irlinger @atmaxinger)\+\_\+ our {\ttfamily Key} and {\ttfamily Key\+Set} datastructures are now fully copy-\/on-\/write! This means noticeably reduced memory usage for cases where keys and keysets are copied and/or duplicated!

We ran some very promising benchmarks, each were performed with 400,000 keys. All benchmarks were executed using {\ttfamily valgrind -\/-\/tool=massif -\/-\/time-\/unit=B -\/-\/max-\/snapshots=200 -\/-\/threshold=0.\+1}.

First, it should be noted that a single key, without counting payload, is about 50\% larger with the copy-\/on-\/write implementation. This explains why the {\ttfamily createkeys.\+c} benchmark yields a negative reduction result. This benchmark only allocates keys, so not much improvement can be expected there. Still, as other stuff also uses heap memory, the overall memory consumption only increased by 22\%, which is far less than 50\%.

All other benchmarks saw meaningful reductions of heap memory used. One interesting observation is that {\ttfamily kdbget.\+c} and {\ttfamily kdbmodify.\+c} used exactly the same memory. This can most likely be explained by internal caching within the memory allocator of {\ttfamily glibc}.

We also performed runtime tests on the same benchmarks using {\ttfamily perf stat -\/-\/repeat 13} to ensure no major performance regressions occur.
