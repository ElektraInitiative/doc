\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1325}{}\doxysection{Problem}\label{doc_decisions_rest_api_documentation_md_autotoc_md1325}
A standard way of describing R\+E\+ST A\+P\+Is offered by tools and plugins for Elektra is required to ease development for and usage of these. Because many good standards for describing A\+P\+Is are out there already, an existing one shall be used.\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1326}{}\doxysection{Constraints}\label{doc_decisions_rest_api_documentation_md_autotoc_md1326}

\begin{DoxyItemize}
\item The chosen standard should support Markdown syntax
\item A\+PI descriptions created within the standard should be human readable
\item Only free software should be used
\item To enhance reusability, there should be a separation of data modeling and A\+PI description
\end{DoxyItemize}\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1327}{}\doxysubsection{Soft-\/\+Constraints}\label{doc_decisions_rest_api_documentation_md_autotoc_md1327}

\begin{DoxyItemize}
\item (Nice looking) documentation should be generateable from the plain documentation
\item Created documentation should be testable by tools (automatic tests)
\end{DoxyItemize}\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1328}{}\doxysection{Assumptions}\label{doc_decisions_rest_api_documentation_md_autotoc_md1328}

\begin{DoxyItemize}
\item There is a well-\/suited standard with enough (free) tools available to satisfy all or most constraints.
\item Scenarios created by Elektra R\+E\+ST A\+P\+Is are simple enough to be allegeable by the chosen standard, also in future.
\end{DoxyItemize}\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1329}{}\doxysection{Considered Alternatives}\label{doc_decisions_rest_api_documentation_md_autotoc_md1329}

\begin{DoxyItemize}
\item \href{http://swagger.io/}{\texttt{ Swagger}}
\item \href{https://apiary.io/}{\texttt{ apiary}} with \href{https://apiblueprint.org/}{\texttt{ A\+PI blueprints}}
\item \href{http://raml.org/}{\texttt{ R\+A\+ML}}
\end{DoxyItemize}\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1330}{}\doxysection{Decision}\label{doc_decisions_rest_api_documentation_md_autotoc_md1330}
The decision is to use \href{https://apiblueprint.org/}{\texttt{ A\+PI blueprints}} together with additional tools from its ecosystem.\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1331}{}\doxysection{Rationale}\label{doc_decisions_rest_api_documentation_md_autotoc_md1331}
{\bfseries{A\+PI Blueprints}} together with some (free) tools for it support all given constraints and also all soft-\/constraints. It also fits the current documentation style of the Elektra Initiative the most.

Additionally to modeling data apart from the A\+PI, {\bfseries{A\+PI Blueprints}} also supports schemata modeling, which is more precise than giving examples for requests and responses.\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1332}{}\doxysection{Implication}\label{doc_decisions_rest_api_documentation_md_autotoc_md1332}

\begin{DoxyItemize}
\item A\+PI descriptions should also follow other conventions like the usage of similar error codes.
\end{DoxyItemize}\hypertarget{doc_decisions_rest_api_documentation_md_autotoc_md1333}{}\doxysection{Notes}\label{doc_decisions_rest_api_documentation_md_autotoc_md1333}
There are many tools available to use with {\bfseries{A\+PI Blueprints}}, for example the \href{https://github.com/apiaryio/apiary-client}{\texttt{ C\+LI tool}} of apiary. Other tools can be found on the \href{https://apiblueprint.org/tools.html}{\texttt{ official website}} of the standard.
